# WP03: Training priorities section

## Objective

Add a "Training Priorities" section to the accessibility report that groups
findings by WCAG success criterion, surfaces the top 5 by total pages-affected,
and flags possible component inconsistencies. Optionally asks Ollama for
plain-English advice.

## Context

- The existing Ollama integration for `ai-findings` is in `src/engines/ollama.js`
  (or similar) — reuse the same connectivity check and request pattern.
- The bug list passed to `report-html.js` already has `wcag_sc`, `rule_id`,
  `pages_affected`, and `severity` — derive the priorities from those.
- The new section belongs above the bug list, below the filter bar, on the
  accessibility report page.
- All training priority content must be present in the generated HTML (no JS
  required to read it).

## Subtasks

### T007: Create src/lib/training-priorities.js

Export a function `computeTrainingPriorities(bugs)` that:
1. Groups bugs by `wcag_sc`.
2. Sums `pages_affected` per SC group.
3. Returns the top 5 SCs by total pages-affected, each with:
   - `wcag_sc`, `label` (plain-English SC name), `total_pages`, `rule_count`,
     `rules` (array of rule_ids in that SC)
4. For each SC where `rule_count >= 3` AND every rule appears on `>= 5` pages,
   sets `component_inconsistency: true`.

Plain-English SC labels must cover at minimum the 20 most common SCs flagged
by axe-core. Use a static lookup table in the same file.

### T008: Unit tests for training-priorities.js

Write `tests/unit/training-priorities.test.js`:
- 10-bug fixture spread across 6 SCs → verify only top 5 returned, correct order.
- 3-rule / same-SC / 6-pages-each fixture → component_inconsistency = true.
- 3-rule / same-SC / 4-pages-each fixture → component_inconsistency = false.
- Single-rule SC → component_inconsistency = false.

### T009: Wire into aggregate.js

In `src/aggregate.js`, after the bug list is built, call
`computeTrainingPriorities(bugs)` and attach the result to the data object
passed to the accessibility report renderer.

If Ollama is configured and reachable (reuse existing check), send a compact
prompt listing the top-5 SC labels and asking for a 2–3 sentence plain-English
training focus statement. Store the response (or `null`) alongside the
priorities object. Fail silently — no error thrown, no report change if Ollama
is absent.

### T010: Render Training Priorities section in report-html.js

Above the bug list (below the filter bar) on the accessibility report page,
render:
1. An `<h2>Training Priorities</h2>` heading.
2. A table with columns: Rank, WCAG SC, Topic, Pages Affected, Rules Firing,
   Flag (renders "Component inconsistency?" when true).
3. If the Ollama summary is non-null, render it in a `<p>` below the table
   with a note that it was generated by AI.

## Validation

Run `npm run test:unit` — all tests must pass.
Run `npm run test:e2e` after wiring aggregate.js.
